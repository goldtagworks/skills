# Blog Writing

사람이 직접 쓴 것처럼 읽히는 한글 블로그 본문을 만드는 스킬이다. SEO 점수가 높아도 AI 냄새가 나면 실패다.

## 메모 입력 스키마 — 글 쓰기 전 필수

글을 시작하기 전에 아래 항목을 확보한다. 항목이 누락되면 작성을 시작하지 않고 먼저 물어본다.

```
실제 있었던 일:
  - 언제, 어디서, 누구랑 (날짜나 상황 설명)

실제 수치/사례/사실:
  - 숫자, 인용, 구체적 사실 (시간, 비용, 성능, 결과 수치 등)
  - 없으면 "없음" 명시

삽질/실패:
  - 안 됐던 것, 헤맸던 것, 틀렸던 것

감정 반응:
  - 놀랐다, 황당했다, 의외였다, 짜증났다 등 (최소 1개)

인용할 만한 말 (있으면):
  - 직접 인용 형태로

이 글의 핵심:
  - 한 줄로 — 읽고 나서 독자가 뭘 가져가길 바라는지
```

"실제 수치/사례"와 "삽질/실패"가 둘 다 없으면 초안을 쓰기 전에 반드시 묻는다: "구체적인 막힌 순간이나 실제 사례가 있었나요? 있으면 그게 글의 뼈대가 됩니다."

## 워크플로우

1. **메모 확인** — 위 입력 스키마 항목 확보 여부 점검. 누락이 있으면 먼저 요청
2. **글의 성격 파악** — 주제, 길이 모드(long/short), 타깃 독자, 톤(평어체/경어체) 확인
3. **톤·구조 결정** — 이 글에 맞는 구조와 호흡을 먼저 정한다. 매번 같은 구조를 쓰지 않는다
4. **통으로 작성** — 도입부터 끝까지 하나의 흐름으로 쓴다. 섹션별로 끊어서 조립하지 않는다
5. **자가 검토** — 글자 수, 키워드 자연스러움, AI 문체 체크 (아래 "금지 패턴" 확인)

## 길이 정책

| Mode | 한글 글자 수 | 언제 |
|---|---|---|
| `long` (기본) | 10000~12000자 | 심층 에세이, 분석, 비교, 회고 |
| `short` | 5000~7000자 | 빠른 팁, 단일 문제 해결, 짧은 노트 |

별도 지정이 없으면 `long`을 기본으로 한다. 작성 도중 임의로 모드를 바꾸지 않는다. **분량을 채우려고 같은 말을 돌려 쓰지 않는다.** 늘려야 하면 구체 사례, 반례, 시나리오, 실제 메모 예시로 깊이를 더한다.

## 출력 형식

저장 경로는 사용자 요청에 따른다(지정이 없으면 어디에 저장할지 묻는다). 마크다운 프론트매터는 아래를 기본으로 하되, 본문 구조는 글마다 다르게 한다.

```markdown
---
title: "{제목}"
slug: "{slug}"
date: "{YYYY-MM-DD}"
description: "{메타 디스크립션, 155자 이내}"
keywords: ["{primary}", "{secondary1}", "{secondary2}"]
lang: "ko"
---

{본문 — 아래 가이드를 따르되, 고정 템플릿 없음}
```

블로그 플랫폼에 프론트매터가 필요 없으면 제목과 본문만 출력해도 된다.

## 사람처럼 쓰기 — 핵심 원칙

### 구조를 매번 다르게

- H2 섹션 수는 글마다 달라야 한다 (3개일 수도, 7개일 수도 있다)
- 모든 글이 "문제 제기 → 구조화 → 1,2,3 정리 → 교훈 → 다음 편 예고" 패턴을 따르면 안 된다
- 가능한 구조 예시 (하나만 쓰지 말 것):
  - 시간순 서사: "이런 일이 있었다 → 이렇게 됐다 → 이게 남았다"
  - 핵심 먼저: 결론부터 던지고 → 왜 그런지 풀어가기
  - 대비 구조: A vs B, 전과 후, 기대와 현실
  - 하나의 장면에서 확장: 구체적 에피소드 → 거기서 파생되는 이야기
  - 질문-탐색: 질문을 던지고 답을 찾아가는 과정 자체를 서술
- "마무리", "정리하며" 같은 결론 섹션을 매번 넣지 않는다. 글이 자연스럽게 끝나면 그냥 끝낸다
- `---` 구분선을 H2 대신 쓰는 것도 가능하다

### 도입부 4박자 호흡 (필수)

블로그 본문 첫 문단은 다음 4박자로 잡는다. 결과부터 던지지 않는다.

1. **동기** — 왜 이걸 하려 했는지 (1인칭 이유)
2. **시도** — 무엇을 했는지 (구체 행동)
3. **의도한 성공** — 목표 도달
4. **의도 안 한 새 문제** — 그런데 다른 데서 깨진 게 있음 (반전)

**예시:**

> 어떻게 하면 돈을 안 들이고 할 수 있을까 고민하다 직접 다 만들어 쓰기로 했다. 그리고 정말 0원이 됐다…

- "0원"이 비용 절감(성공)과 들인 시간·품(새 문제) 두 의미로 동시에 읽힌다
- 줄임표(…)가 3→4 전환점에서 호흡을 만든다

**적용 규칙:**

- 첫 문장은 동기 또는 장면. **통계·결론·결과로 시작하지 않는다**
- 줄임표는 4박자 호흡 전환에만 쓰고 남발하지 않는다
- 가능하면 같은 단어·표현이 성공과 문제 두 의미를 동시에 가리키게 한다
- 섹션 단위에도 응용 가능: "표면 이유 → 진짜 욕망", "통념 → 데이터로 깨기", "분류 → 보정"

### 문장 리듬을 불균질하게

- **짧은 문장과 긴 문장을 섞는다.** 3~5줄 문단만 반복하면 AI 티가 난다
- 한 줄짜리 문단을 의도적으로 쓴다. "그래서 포기했다." 같은 짧은 문단은 리듬을 깬다
- 반대로 6~8줄로 길게 풀어쓰는 문단도 있어야 한다. 모든 문단이 같은 길이면 기계적이다
- 문장 시작을 다양하게: "그런데", "솔직히", "아, 그리고", "이건 좀", "근데 이게" 같은 구어체 시작도 섞는다
- 완결되지 않는 문장, 말줄임표를 활용한다
- 한국어 본문에서 em-dash(—)를 문장 연결에 쓰지 않는다. 보충 설명은 괄호나 마침표로 분리한다

### 감정과 판단을 드러내기

- "~했다"의 객관적 서술만 하지 않는다. "좀 황당했다", "이건 진짜 좋았다", "솔직히 기대 이하였다" 같은 주관적 반응을 넣는다
- 실패, 삽질, 시행착오를 숨기지 않는다. 깔끔하게 정리된 성공담만 쓰면 AI 같다
- "나"의 판단 근거를 쓴다: "그렇게 했다"보다 "이렇게 할 수도 있었는데, 내가 이걸 선택한 이유는..."

### 구체성으로 승부

- 추상적 요약 대신 구체적 디테일: "여러 번 시도했다" → "세 번째 시도에서 결과가 달라졌다"
- 실제 인용, 숫자, 대화 발췌, 장면 묘사, (해당되면) 코드·에러 메시지를 넣는다
- 수치를 쓴다: "빨라졌다" → "3.2초에서 0.8초로 줄었다"
- Before/After 비교가 가능하면 보여준다

### 설명을 끝까지 펼쳐라

"설명하려다 마는" 글은 AI 티가 강하게 난다. 하나의 주장이나 현상을 도입했으면, 최소 한 단계 더 풀어내고 넘어간다.

- **현상만 언급하고 지나가지 않는다**: 어떤 현상을 한 줄로 짚고 끝내면 피상적이다. 왜 그게 문제인지, 어떤 부작용이 쌓이는지, 어떤 모습으로 드러나는지까지 이어서 쓴다
- **추상적 개념은 구체적 시나리오로 펼친다**: 명제는 실제로 그 개념이 필요한 순간을 예시로 보여줘야 살아난다
- **한 줄로 끝낼 수 있는 말이면 도입하지 마라**: 도입했으면 최소 3~5줄로 완결한다. 호흡이 짧게 끊기면 독자가 흡수하지 못하고 지나간다
- **"예를 들어"를 아끼지 마라**: 추상 → 구체 → 다시 추상의 리듬이 독자의 뇌에 그림을 그려준다
- **새 용어를 던지고 방치하지 마라**: 낯선 개념을 꺼냈으면 반드시 한 문단 안에서 의미와 예시를 같이 제시한다

## 금지 패턴 — AI 냄새 나는 표현/구조

다음 패턴이 글에 나타나면 반드시 고친다.

### 금지 표현 (직접적으로 쓰지 말 것)
- "정리하면", "정리해보면", "핵심은", "결론적으로", "요약하자면"
- "첫째/둘째/셋째", "첫 번째/두 번째/세 번째" 나열. 목록이 필요하면 불릿 리스트나 자연어로 풀어 쓴다
- "~하는 것이 중요하다", "~할 필요가 있다". 누구한테 강의하는 톤이 된다
- "방법론 N가지", "핵심 포인트 N개"
- "다음 편에서 계속", "다음 편에서 다루겠다". 시리즈라도 매번 쓰지 않는다
- "독자 여러분", "여러분은 어떠신가요?"
- 영어 분류 용어를 그대로 던지기("Out of Scope", "Source of Truth", "ground truth" 등). 한국어로 풀어 쓰거나, 꼭 필요하면 한국어 본문에 영어 병기를 한 번만
- "정직해진다", "도달 가능", "트레이드오프 발생", "포지셔닝" 같은 추상 명사 남발. 구체 동작이나 상황으로 바꿔 쓴다
- "라는 컨셉을 표방한다", "라는 컨셉을 진지하게 받아들이려면". 그냥 "X를 목표로 한다"로
- "결론부터 말하면 X. 그 이유는 Y" 같은 자기 대화 패턴. 한국어 본문은 결론을 자연스럽게 흐름 안에 둔다

### 금지 문장 구조 (영어 직역체)

영어 글쓰기에서 흔한 패턴이 한국어에 들어오면 바로 AI 냄새가 난다.

**em-dash로 명사·서술 잇기 (`X — Y`)**
- 잘못: `이 방식이 좋다 — 관리가 편해서`
- 좋음: `이 방식이 좋다. 관리가 편하기 때문이다`
- 한국어는 조사(은/는, 이/가)로 잇거나 두 문장으로 끊는다. em-dash는 본문에서 거의 쓰지 않는다(격식 문구 정도만)

**헤딩·제목의 괄호 부가설명 (`X (Y)`)**
- 잘못: `자주 쓰는 명령 (처음 세팅한 뒤에)`
- 좋음: `처음 세팅한 뒤에 자주 쓰는 명령` 또는 부가 정보는 본문에 푼다
- 한국어 헤딩은 수식어를 앞에 붙여 한 구로 만들거나, 부가 정보는 본문에 푼다

**가운뎃점(`·`) 남발**
- 잘못: `노트북·태블릿·휴대폰`, `구조·관리·전달`
- 좋음: `노트북, 태블릿, 휴대폰`, `구조와 관리, 전달`
- 둘 사이는 "와", 셋 이상은 ", "로 풀어 쓴다

**이모지·화살표 본문 안에서**
- 잘못: 본문에 `🟢 시작 전에`, `→ 다음 단계`, `✅❌🟢🔴` 같은 분류 이모지·화살표가 자주 등장
- 좋음: 표나 체크리스트에서 가끔만. 본문 산문에서는 자연어로

### 금지 구조
- 모든 H2 섹션이 비슷한 길이. 한 섹션은 3줄, 다른 섹션은 15줄. 이런 불균형이 자연스럽다
- "문제 → 해결 → 교훈"이 3번 이상 반복되는 글
- 매 섹션 끝에 교훈/인사이트 한 줄씩 정리하는 패턴
- 글의 마지막이 항상 "깔끔한 마무리 + CTA"인 것
- 한 문서 안에서 평어체(~다)와 경어체(~합니다)를 섞기. 글의 톤을 정하면 끝까지 일관되게 간다

## SEO 규칙 (유연하게 적용)

키워드 배치는 가이드라인이지 공식이 아니다. 자연스러움이 우선.

- Primary Keyword: 제목 + **첫 100단어 이내** 자연스럽게 포함 (흐름을 해치지 않는 선에서)
- Secondary Keyword: H2 제목 일부에 포함하되, 모든 H2에 넣지 않는다
- 키워드 밀도: 1~2% 목표이되, 어색하게 느껴지면 밀도를 낮춰도 된다
- 동의어, 관련 표현, 일상 용어를 섞어서 키워드가 기계적으로 반복되지 않게 한다
- featured snippet 노림: 핵심 답변을 상단에 배치하되, 글 전체 흐름을 해치지 않는 선에서
- 외부 링크 2~3개: 주장이나 데이터에 근거가 되는 신뢰 있는 외부 출처를 자연스럽게 연결

## 제목 작성 규칙 (필수)

제목은 **검색 매칭**과 **클릭 유인** 두 가지를 모두 만족해야 한다. 소재를 그대로 옮긴 설명형 제목("X란 무엇인가")은 클릭률이 낮다.

### 제목 공식

`(소재/주제) + (구체 키워드) + (어그로 장치 1개)` 형태로 조립한다.

**어그로 장치 (1개 이상 필수) — 강도 6.5~7 기준**

신뢰를 깨지 않는 강도에서 멈춘다. 유튜브 썸네일급 과장(`~는 죽었다`, `무조건 해야 한다`, `아무도 안 쓴다`)은 금지.

1. **숫자 + 결과** — `10분 만에`, `3단계로`, `0원으로`, `40% 올리기`
2. **기존 상식 반박** — `~가 아니다`, `~는 필요 없을까`, `~보다 더 중요한 것`
3. **경고/실수 방지** — `~하기 전에 먼저 알아야 할 것`, `~의 함정`, `~에서 가장 많이 틀리는 N가지`
4. **비교/선택 강제** — `A vs B`, `~에서 반드시 버려야 하는 것`
5. **시행착오 공개** — `직접 해보니 ~가 더 중요했다`, `~가 생각보다 어려운 이유`, `먼저 막힌 지점들`

**톤 강도 기준**
- 권장 강도 = 6.5~7
- 금지 과장: `~는 죽었다`, `무조건 해야 한다`, `아무도 안 쓴다`, `~는 끝났다`

### 제목 자가검증 (작성 후 반드시 실행)

제목을 확정하기 전에 아래 질문을 통과해야 한다:
1. **소재가 제목에서 바로 보이는가?** — 무슨 글인지 한눈에 들어오는가
2. **어그로 장치가 있는가?** — 단순 설명형 `~를 소개한다`, `~에 대하여`는 실패
3. **구체적 숫자나 결과가 있는가?** (권장, 필수 아님) — 있으면 CTR이 훨씬 오른다

1번과 2번은 필수. 통과 못 하면 제목 후보 3개 더 생성하고 재평가한다.

### 제목 전환 예시

| Before (설명형) | After (소재 + 키워드 + 어그로) |
|---|---|
| 홈카페 입문 | 홈카페 입문 — 1만 원으로 카페 라떼 따라잡는 법 |
| 제주도 여행기 | 제주도 3박 4일 — 렌터카 없이 다녀보고 깨달은 것 |
| 부업 후기 | 6개월 부업해서 번 돈 공개 — 생각보다 안 남는 이유 |
| 운동 루틴 정리 | 헬스 1년차가 가장 많이 틀리는 3가지 자세 |

## 기본 규칙

- 총 글자 수는 위 **길이 정책** 표를 따른다
- 능동태 우선
- 주요 주장에 출처 포함 (단, 매 문장마다 넣지 않는다. 핵심 데이터·통계에만)
- 도입부는 독자를 끌어당기되, "3줄 이내" 같은 길이 제한 없음. 한 줄이어도 되고 다섯 줄이어도 된다

## 에러 처리
- 글자 수 미달/초과: 목표 범위에 맞게 조정
- 키워드 부자연스러움: 키워드를 줄이더라도 자연스러움을 우선한다
- AI 문체 감지: 위 금지 패턴에 해당하면 즉시 리라이트
